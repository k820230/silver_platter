# 퀀트 기반 주식 자동매매 프로그램 요구사항 정의서

- 작성일: 2026-05-22

- 저장 위치: `/doc/silver_platter`
- 기반 자료: `/doc/codex_log_jhkim5`의 투자 관리, 퀀트 준비, 리스크 관리, 검증, 데이터 파이프라인, API, DBMS, 서버 구성 조사 문서

## 1. 문서 목적

이 문서는 개인 또는 소규모 운용자가 사용할 퀀트 기반 주식 자동매매 프로그램의 초기 요구사항을 정의한다.

목표는 단순한 종목 추천기나 수익률 최대화 엔진이 아니라, 다음을 갖춘 투자 운영 시스템을 만드는 것이다.

- 국내/해외 금융상품 데이터를 수집하고 정합성을 검증한다.
- 투자 가설, 전략, 모델, 백테스트, 리스크 한도, 주문 결과를 추적 가능하게 관리한다.
- 리스크 한도 안에서 리밸런싱 후보와 주문 후보를 생성한다.
- 초기에는 수동 승인과 모의운용을 중심으로 운영하고, 검증 후 제한적으로 자동 주문을 허용한다.
- 모든 주문, 체결, 잔고, 모델 판단, 리스크 차단 결과를 감사 가능하게 기록한다.

이 문서는 투자, 세무, 법률 자문이 아니다. 세법, 거래소 규정, 데이터 라이선스, 증권사 API 조건은 변경될 수 있으므로 별도 검증과 기준일 관리가 필요하다.

## 2. 프로젝트 범위

### 2.1 MVP 범위

MVP는 자동매매보다 데이터와 리스크 통제의 신뢰성을 우선한다.

- 국내와 미국 상장 주식, ETF, ETN 중심의 종목 마스터 관리
- 계좌, 포트폴리오, 거래 원장, 현금 원장 관리
- 일봉 기반 historical backfill
- 환율, 공시, 재무, 기업 이벤트의 기본 수집
- 종목별 팩터와 리스크 지표 계산
- 전략별 후보 종목 점수화
- 백테스트 및 시나리오 테스트
- 주문 후보 생성
- 주문 전 리스크 체크
- 수동 승인 기반 주문 또는 paper trading
- Web UI와 모바일 브라우저 접근
- 운영 로그, 감사 로그, 데이터 품질 리포트

### 2.2 MVP 이후 확장 범위

- 분봉 및 실시간 체결/호가 수신
- Redis Streams 기반 실시간 이벤트 처리
- 장중 포트폴리오 평가와 리스크 알림
- 한국투자증권 Open API 기반 국내/미국 실거래 API 연동
- 제한 금액, 제한 상품, 제한 전략에 대한 자동 주문
- Goldilocks 파티셔닝/scale-out, ClickHouse, QuestDB 등 시계열/대용량 저장소 확장
- 옵션, 선물, 채권, 암호자산 등 멀티에셋 확장

### 2.3 초기 제외 범위

- 고빈도 매매
- 초저지연 체결 시스템
- 무제한 자동 주문
- 모델 신호만으로 리스크 체크를 우회하는 주문
- 비상장주식 자동매매
- 레버리지, 공매도, 옵션 매도 등 고위험 전략의 자동 실행
- 기관급 멀티유저 주문 관리 시스템
- Interactive Brokers 등 별도 해외 브로커 API 연동

비상장주식은 초기에는 자동매매 대상이 아니라 가격, 증빙, 유동성, 리스크, 세금 보조정보 관리 대상으로 둔다.

## 3. 핵심 설계 원칙

1. 리스크는 하드 제약이다.
   수익 모델의 신호가 강해도 리스크 한도를 넘으면 주문하지 않는다.

2. 수익은 리스크 제약 안에서 최적화한다.
   목적함수는 세후, 비용 후, 리스크 조정 수익률이어야 한다.

3. 데이터 품질이 전략 품질이다.
   원본 데이터, 정규화 데이터, 수정 데이터, 수집 시각, 제공자, 라이선스를 분리해 관리한다.

4. 자동매매는 마지막 단계다.
   분석, 주문 후보, 수동 승인, 모의운용, 소액 실거래 검증을 거친 뒤 제한적으로 적용한다.

5. 모든 판단은 재현 가능해야 한다.
   어떤 데이터 버전, 모델 버전, 전략 파라미터, 리스크 규칙으로 판단했는지 기록한다.

6. 주문 전 차단 규칙은 필수다.
   계좌 잔고, 주문 수량, 가격 괴리, 유동성, 포트폴리오 한도, 중복 주문을 확인한다.

7. 운영 장애 시 안전하게 멈춘다.
   데이터 지연, API 장애, 대사 불일치, 주문 실패가 발생하면 자동 주문을 중단한다.

## 4. 사용자와 역할

### 4.1 개인 투자자/운용자

- 포트폴리오 상태를 확인한다.
- 전략 신호와 주문 후보를 검토한다.
- 수동 승인 또는 거부를 수행한다.
- 리스크 경고와 데이터 품질 이슈를 확인한다.

### 4.2 퀀트 리서처

- 전략 가설을 등록한다.
- 피처, 팩터, 모델을 실험한다.
- 백테스트, 워크포워드, 시나리오 테스트 결과를 비교한다.
- 실패한 실험도 기록한다.

### 4.3 시스템 관리자

- API 키, 데이터 라이선스, 서버, DB, 백업, 모니터링을 관리한다.
- 데이터 수집 실패, DB 장애, 주문 API 장애에 대응한다.
- 배포와 운영 로그를 관리한다.

## 5. 주요 업무 흐름

### 5.1 데이터 수집 흐름

```text
Provider API / Bulk Download / WebSocket
  -> Raw Data Store
  -> Normalizer
  -> Canonical Store
  -> Data Quality Checker
  -> Research Store
  -> Strategy / Risk / UI
```

필수 조건:

- 외부 API 응답은 원본 형태로 먼저 저장한다.
- 내부 표준 스키마로 정규화하기 전에는 원본을 덮어쓰지 않는다.
- 실시간 데이터와 장 종료 후 공식 데이터를 구분한다.
- 데이터 품질 오류가 심각하면 전략 신호와 주문 후보 생성을 중단한다.

### 5.2 전략 연구 흐름

```text
투자 가설 등록
  -> 데이터셋 생성
  -> 팩터/피처 계산
  -> 모델 학습 또는 규칙 기반 전략 실행
  -> 백테스트
  -> 워크포워드 검증
  -> 시나리오 테스트
  -> 모의운용 후보 등록
```

필수 조건:

- 전략은 설명 가능한 투자 가설을 가져야 한다.
- 실패한 실험도 기록한다.
- 백테스트 결과만으로 실거래를 허용하지 않는다.
- 단순 벤치마크와 기준 모델보다 비용 후 성과 또는 리스크 관리 효과가 있어야 한다.

### 5.3 주문 생성 및 실행 흐름

```text
전략 신호
  -> 리밸런싱 후보
  -> 주문 후보
  -> 주문 전 리스크 체크
  -> 수동 승인 또는 자동 승인 정책
  -> broker API 주문
  -> 체결/부분체결/취소 반영
  -> 계좌/현금/포지션 대사
```

필수 조건:

- 초기 운영은 주문 후보 생성과 수동 승인 중심으로 한다.
- paper trading을 실거래와 분리한다.
- 주문 실패, 부분체결, API 장애, 중복 요청에 대한 상태 기계를 구현한다.
- 실제 체결 후 포트폴리오, 현금, 수수료, 세금, 리스크 지표를 다시 계산한다.

### 5.4 검증 및 출시 흐름

```text
단위 테스트
  -> 계산 로직 검산
  -> 데이터 품질 검증
  -> 백테스트 검증
  -> 시나리오 테스트
  -> 모의운용
  -> 소액 실거래
  -> 제한적 자동화
```

Go/No-Go 기준:

- 요구사항과 사용 범위가 문서화되어 있다.
- 리스크 한도와 중단 조건이 숫자로 정의되어 있다.
- 데이터 장애 시 주문 후보를 만들지 않는다.
- 주문 전 차단 규칙이 정상 작동한다.
- 모의운용과 소액 실거래에서 계좌 대사가 일치한다.

#### 6.2.1 사업 그룹 분류 및 그룹 리스크

사업 그룹은 섹터나 거래소 산업분류보다 더 실무적인 단위로 관리한다. 예를 들어 `메모리 반도체`, `파운드리`, `AI 반도체`, `2차전지 소재`, `전력망 장비`, `클라우드 인프라`, `사이버보안`, `자동차 OEM`, `조선`, `방산`, `바이오 CDMO`처럼 실제 사업 충격이 함께 전파될 수 있는 종목군을 뜻한다.

그룹 분류 원칙:

- 국내와 해외 종목을 같은 사업 기준으로 묶는다.
- 거래소/국가/통화/섹터와 독립적인 내부 `business_group_id`를 둔다.
- 표준 산업분류는 보조 기준으로 사용하고, 실제 매출원, 제품, 공급망, 고객군, 경쟁관계를 반영한다.
- ETF는 구성종목을 풀어헤쳐 사업 그룹 노출을 계산한다.
- 그룹 매핑은 자동 분류 결과와 수동 검토 결과를 구분한다.
- 분류 변경 이력을 보존해 과거 시점의 그룹 노출을 재현할 수 있어야 한다.

그룹 리스크 평가 항목:

- 그룹별 평가금액과 포트폴리오 비중
- 그룹별 국가, 통화, 거래소, 시가총액, 유동성 분포
- 그룹별 변동성, beta, MDD, VaR, CVaR
- 그룹 내부 종목 간 상관관계와 동시 하락 위험
- 그룹 간 상관관계와 숨은 집중도
- 그룹별 평균 거래대금, 호가 스프레드, 예상 청산 소요일
- 그룹별 실적 하향, 밸류에이션 부담, 뉴스/공시 이벤트, 규제/정책 리스크
- 같은 사업을 영위하는 국내외 peer 대비 상대성과와 급격한 괴리

#### 6.6.1 종목별 매매 기록과 FIFO 손익 관리

종목별 매매 기록은 브로커 체결 데이터와 프로그램 원장을 대사할 수 있는 기준 데이터로 관리한다.

FIFO 매칭 원칙:

- 매수 체결은 계좌, 종목, 통화, 체결 시각, 체결 id 기준으로 position lot을 생성한다.
- 매도 체결은 같은 계좌와 종목의 미청산 매수 lot 중 가장 오래된 lot부터 순서대로 소진한다.
- 부분 매도 또는 부분체결은 매도 체결 하나가 여러 매수 lot에 나뉘어 연결될 수 있다.
- 수수료와 세금은 체결 단위 원장을 보존하되, FIFO 매칭 손익 계산에는 매칭 수량 비율로 배분한다.
- 실현손익은 gross P&L, net P&L, 거래 통화 손익, 원화 환산 손익을 분리해 저장한다.
- 남아 있는 매수 lot은 미실현손익, 보유 기간, 평균단가, 세금 리포트의 입력으로 사용한다.
- FIFO 매칭 결과는 재현 가능해야 하며, 산식 또는 세율 변경 시 calculation_version을 바꿔 재계산 이력을 남긴다.

- 해외 주식 양도차익은 원화 기준 양도가액에서 취득가액과 필요경비를 차감해 계산한다.
- 국내/국외 과세대상 주식 양도손익은 같은 과세기간에 손익통산할 수 있다.
- 기본공제는 국내/국외 주식 합산 연 2,500,000원을 기본값으로 둔다.

표시 원칙:

#### 6.7.1 사용자 지정 종목 ML 분석

사용자는 관심 종목, 보유 종목, 후보 종목, 사업 그룹 내 특정 종목을 선택해 ML 분석 작업을 실행할 수 있어야 한다.

예측 대상:

- 주식가격: 다음 거래일/1주/1개월 수익률, 예상 가격 범위, 상승/하락 확률
- 거래량: 평균 대비 거래량 증가 확률, 거래량 급증, 유동성 악화 가능성
- 변동성: 실현 변동성, 예측 변동성, 변동성 급등 확률
- 위험도: 예상 낙폭, VaR/CVaR, 손실 확률, 유동성 리스크, 이벤트 리스크

분석 관리 항목:

- 분석 대상 종목과 기간
- 예측 horizon
- 사용한 데이터 범위와 feature set
- 사용한 모델 종류와 모델 버전
- 학습/검증/테스트 기간
- 예측 생성 시각과 사용 가능 시각
- 예측값, 신뢰구간, confidence score
- 실제 결과와 예측 오차
- 모델 성능 저하 여부

표시 원칙:

- 가격 예측은 단일 목표가가 아니라 범위와 확률로 표시한다.
- 거래량과 변동성 예측은 리스크와 체결 가능성 판단에 연결한다.
- 위험도 예측은 기존 리스크 엔진의 한도 판단보다 낮은 우선순위를 가진다.
- 예측 결과에는 모델 한계, 최근 성능, 데이터 품질 상태를 함께 표시한다.
- 사용자 지정 분석 결과가 강한 매수 신호를 보여도 주문 전 리스크 체크를 우회할 수 없다.

#### 6.10.1 종목별 변동성 지수 및 위험도 지수

VIX처럼 시장의 기대 변동성을 직접 복제하기 어렵더라도, 개별 종목 단위의 위험 상태를 일관되게 비교할 수 있는 내부 지수를 만든다.

종목별 변동성 지수 입력 후보:

- 실현 변동성
- EWMA 변동성
- GARCH 또는 ML 기반 예측 변동성
- 장중 고저가 변동폭
- 거래량 급증
- 옵션 데이터가 있는 경우 implied volatility
- 시장/사업 그룹 대비 상대 변동성

종목별 위험도 지수 입력 후보:

- 예상 낙폭
- VaR/CVaR
- 최대낙폭과 회복 기간
- 유동성 점수
- 헤드라인/공시 이벤트 리스크
- 사업 그룹 동시 하락 위험
- 환율, 금리, 원자재 등 외부 민감도

표시 원칙:

- 지수는 0~100 또는 등급형으로 정규화해 종목 간 비교가 가능해야 한다.
- 원지표와 정규화 지수를 모두 보존한다.
- 지수 산식은 버전 관리하며, 과거 지수를 새 산식으로 덮어쓰지 않는다.
- 지수 변화는 주문 전 리스크 체크, 알림, 수동 검토에 연결하되 자동 주문을 직접 발생시키지 않는다.

사업 그룹 변동성 비교 원칙:

- 사용자가 종목 그룹 리스트에서 복수 사업 그룹을 선택할 수 있어야 한다.
- 사용자가 지정한 기준일의 그룹 변동성값을 기준값으로 사용한다.
- 비교 그래프는 `((비교일 변동성 / 기준일 변동성) - 1) * 100` 형태의 변동 %를 기본으로 표시한다.
- 기준일 변동성값이 0, 결측, 비정상값이면 해당 그룹을 비교 대상에서 제외하거나 별도 경고를 표시한다.
- 기준일 이후 편입 종목 변경이 있는 경우 동일 구성 기준과 당시 구성 기준을 선택할 수 있어야 한다.
- 그룹 변동성은 그룹 내 종목의 시가총액, 보유 비중, 동일 가중 중 하나를 선택해 계산할 수 있어야 한다.

### 9.4 Goldilocks 적용 기준

기준 DBMS는 PostgreSQL이 아니라 현재 작업 디렉토리에 있는 Goldilocks를 사용한다.

확인된 로컬 구성:

```text
Goldilocks DBMS source/runtime: /home/jhkim5/work/product
Python DB API driver: Gliese/home/app_dev/pygoldilocks
Async Python driver: Gliese/home/app_dev/aiogoldilocks
SQLAlchemy 2.0 dialect: Gliese/home/app_dev/SQLAlchemy/2.0/sqlalchemy_goldilocks
Hibernate dialect: Gliese/home/app_dev/Hibernate
ODBC/ESQL libraries: layer/GlieseLibrary
```

Python/FastAPI 계층의 기본 연결 방식:

```python
from sqlalchemy import create_engine

engine = create_engine("goldilocks://user:password@127.0.0.1:22581")
```

Goldilocks 사용 전 준비 요구사항:

- `GOLDILOCKS_HOME` 환경변수를 Goldilocks 설치 경로로 설정한다.
- Goldilocks listener와 DB service의 기동 방식, 접속 host, port, 계정, 권한을 운영 환경변수로 분리한다.
- 초기 Docker Compose 환경에서는 Goldilocks를 현재 디렉토리의 호스트 DB 서비스로 연결하거나, 별도 Goldilocks service 주소로 연결할 수 있게 설계한다.
- `pygoldilocks`, `aiogoldilocks`, `sqlalchemy-goldilocks`를 프로젝트 가상환경에 설치한다.
- SQLAlchemy 모델은 가능한 소문자 identifier를 사용한다.
- 자동 증가 키는 Goldilocks의 `IDENTITY` 또는 `SEQUENCE` 방식으로 설계한다.
- Alembic migration은 Goldilocks dialect에서 생성되는 DDL을 검증한 뒤 운영 DB에 적용한다.
- PostgreSQL 전용 타입, 확장, 함수, TimescaleDB hypertable 같은 기능에 의존하지 않는다.
- 시계열 데이터 증가 시 우선 Goldilocks 파티셔닝/scale-out을 검토하고, tick/orderbook급 대용량은 ClickHouse 또는 QuestDB 보조 저장소로 분리한다.

- Docker Compose 개발 환경
- Goldilocks, Redis, FastAPI, React/Vite 기동
- Alembic migration 체계
- 종목 마스터, 계좌, 포트폴리오, 거래 원장 기본 스키마

완료 기준:

- Web/API/DB/worker가 로컬에서 실행된다.
- 모바일 브라우저에서 Web UI에 접근할 수 있다.
- 기본 migration과 health check가 동작한다.

### 10.2 2단계: Historical Data Pipeline

목표:

- provider 등록
- 종목 universe 수집
- 일봉 backfill
- raw data store
- normalizer
- data quality checker
- Parquet export

완료 기준:

- 특정 시장/종목군의 일봉 데이터를 재현 가능하게 수집한다.
- 결측/중복/비정상 가격 리포트가 생성된다.
- 수집 실패 후 checkpoint 기반 재시작이 가능하다.

### 10.3 3단계: 포트폴리오와 리스크 엔진

목표:

- 거래 원장, 현금 원장, 보유 로트 계산
- 종목별 매매 기록과 FIFO 매수 lot-매도 체결 매칭
- 환율 반영 평가
- 실현/미실현 손익
- 변동성, MDD, VaR/CVaR, 집중도, 유동성 지표
- 국내외 유사 사업 그룹 노출과 그룹 리스크 지표
- 리스크 한도 설정

완료 기준:

- 샘플 거래내역의 수량, 평균단가, FIFO lot 매칭, 현금, 실현/미실현손익이 수동 계산과 일치한다.
- 리스크 한도 초과 시 주문 후보가 차단된다.

### 10.4 4단계: 백테스트와 시나리오 테스트

목표:

- point-in-time 백테스트
- 비용, 세금, 환전, 슬리피지 반영
- 벤치마크 비교
- 시장 충격, 데이터 장애, 주문 실패 시나리오

완료 기준:

- 생존편향과 미래정보 누수 방지 규칙이 테스트로 검증된다.
- 주요 스트레스 시나리오에서 손실, 차단, 알림 결과가 리포트로 나온다.

### 10.5 5단계: 전략 신호와 주문 후보

목표:

- 전략 등록
- 팩터 계산
- 후보 종목 점수화
- 목표 비중 계산
- 주문 후보 생성
- 주문 전 리스크 체크

완료 기준:

- 전략 신호가 주문 후보로 생성되지만, 수동 승인 전에는 실거래 주문이 나가지 않는다.
- 주문 후보에는 근거 신호와 리스크 체크 결과가 포함된다.

### 10.6 6단계: Paper Trading

목표:

- paper 계좌
- 클라이언트 실시간 테스트 모드와 가상 계좌 생성/초기화
- 모의 주문/체결
- 실시간 시장 데이터 기반 가상 체결 엔진
- 체결 후 포지션/현금 반영
- FIFO 손익과 해외 주식 양도소득세 예상 반영
- 모의 대사 리포트

완료 기준:

- 최소 3개월, 가능하면 6개월 이상 신호, 주문 후보, 리스크 알림, 리포트가 정상 생성된다.
- 클라이언트에서 가상 계좌로 주문을 실행해도 실제 broker API 호출이 발생하지 않는다.
- 가상 체결 후 포지션, 현금, FIFO 손익, 해외 주식 예상세액이 실시간 갱신된다.
- 데이터 장애 시 주문 후보가 생성되지 않는다.

### 10.7 7단계: 소액 실거래와 제한적 자동화

목표:

- 브로커 API 연동
- 실거래 주문/취소/체결 상태 처리
- 부분체결 처리
- 계좌 대사
- kill switch
- 제한 금액 자동화

완료 기준:

- 실거래 계좌 잔고와 프로그램 잔고가 일치한다.
- 주문 실패와 부분체결에서 중복 주문이 발생하지 않는다.
- 자동 주문은 제한된 전략, 상품, 금액에서만 작동한다.

## 11. 테스트 요구사항

### 11.1 단위 테스트

- 수익률 계산
- 평균단가 계산

- 매수 lot과 매도 체결의 FIFO 매칭
- 부분 매도와 여러 매수 lot 분할 매칭
- FIFO 기준 실현손익과 보유 기간 계산
- 해외 주식 연간 양도소득세 예상 계산
