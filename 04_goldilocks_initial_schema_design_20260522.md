## 1. 문서 목적

이 문서는 퀀트 기반 주식 자동매매 프로그램의 Goldilocks 초기 스키마 설계 기준을 정의한다. 목표는 MVP 구현 전에 테이블 생성 순서, key 전략, index 전략, migration 기준, seed data, 검증 절차를 정리하는 것이다.

이 문서는 실제 운영 DDL의 1차 설계안이다. Goldilocks 로컬 인스턴스에서 DDL smoke test를 수행한 뒤 세부 문법, index 옵션, partitioning 옵션을 확정한다.

## 2. 적용 범위

초기 스키마는 아래 범위를 포함한다.

- 종목/시장 마스터
- provider, 라이선스, 수집 manifest
- 가격, 재무, 기업 이벤트
- 공시/뉴스/국제 정세 이벤트
- 전략, 모델, 신호, 백테스트
- 계좌, 포트폴리오, 거래/현금 원장
- FIFO lot 매칭과 실현손익
- 해외 주식 양도소득세 예상
- 주문, 체결, simulation 체결
- 리스크 지표, 한도, 주문 전 체크
- DB 백업/복구 이력
- 감사 로그와 의사결정 로그

대량 tick/orderbook 원천 저장, 고빈도 호가 이력, 대규모 모델 feature matrix는 초기 Goldilocks 스키마가 아니라 DuckDB + Parquet 또는 향후 ClickHouse/QuestDB로 분리한다.

## 3. Goldilocks DDL 기준

### 3.1 기본 식별자 규칙

| 항목 | 규칙 |
| --- | --- |
| schema | 초기에는 애플리케이션 접속 계정 기본 schema 사용. 별도 schema 이름은 `quant` 후보 |
| table | `snake_case` 소문자 |
| column | `snake_case` 소문자 |
| primary key | `{table_singular}_id` 또는 명시적 domain id |
| foreign key | `{referenced_table_singular}_id` |
| unique index | `uq_{table}_{column_or_purpose}` |
| normal index | `ix_{table}_{column_or_purpose}` |
| check constraint | `ck_{table}_{purpose}` |
| foreign key constraint | `fk_{child}_{parent}` |

### 3.2 타입 매핑

| 논리 타입 | Goldilocks DDL 후보 | 사용 예 |
| --- | --- | --- |
| bigint id | `BIGINT GENERATED BY DEFAULT AS IDENTITY` | surrogate PK |
| bigint FK | `BIGINT` | FK |
| code | `VARCHAR(32)` | 상태, 유형, 통화 |
| short text | `VARCHAR(255)` | 이름, 제목 |
| long text | `LONG VARCHAR` | 설명, error message, 요약 |
| decimal money | `DECIMAL(28, 10)` | 가격, 금액 |
| decimal ratio | `DECIMAL(18, 10)` | 수익률, 비중 |
| volume quantity | `DECIMAL(28, 8)` | 수량, 거래량 |
| date | `DATE` | 영업일, 결제일 |
| timestamp | `TIMESTAMP` | event, receive, loaded |
| boolean | `BOOLEAN` | flags |

Goldilocks SQLAlchemy dialect와 Alembic에서 type rendering을 확인한 뒤 `DECIMAL`, `NUMBER`, `TIMESTAMP WITH TIME ZONE` 사용 범위를 최종 확정한다. MVP에서는 timezone을 별도 `timezone_code` 또는 UTC 저장 규칙으로 통제하고, `TIMESTAMP`를 기본으로 둔다.

### 3.3 공통 컬럼 template

기준 테이블 template:

```sql

updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL,
```

## 7. Index 설계

### 7.1 공통 index 원칙

- FK 컬럼에는 조회 경로가 있으면 index를 둔다.
- 시간순 조회가 많은 테이블은 `{domain_id}, {event_ts}` 또는 `{domain_id}, {created_at}` 복합 index를 둔다.
- 원장과 감사 로그는 account/order 중심 index를 둔다.
- 대량 가격 데이터는 `security_id + interval + time` unique index를 기준으로 한다.
- low cardinality status 단독 index는 피하고, 주요 조회 조건과 묶는다.

### 10.2 migration 작성 원칙

- Alembic autogenerate 결과를 그대로 믿지 않고 Goldilocks dialect DDL을 직접 검토한다.
- 긴급 수정 외에는 하나의 migration에 너무 많은 domain 변경을 섞지 않는다.
- 대량 테이블 생성 migration과 index 생성 migration을 분리한다.
- 운영 테이블 drop은 금지하고 deprecate 후 별도 정리 절차를 둔다.
- migration 실행 전 DB 백업 상태를 확인한다.

## 11. SQLAlchemy 모델 기준

### 11.1 연결

```python
from sqlalchemy import create_engine

engine = create_engine("goldilocks://user:password@127.0.0.1:22581")
```

### 11.2 모델 작성 규칙

- `BigInteger` PK와 `Identity` 또는 dialect 호환 identity를 사용한다.
- `String(length)`에는 반드시 length를 지정한다.
- 금액/가격은 `Numeric(28, 10)`을 기본으로 한다.
- 수량은 `Numeric(28, 8)`을 기본으로 한다.
- 시간은 `DateTime`을 사용하고 UTC 저장 규칙을 애플리케이션에서 강제한다.
- enum은 DB enum이 아니라 `VARCHAR` + application validation으로 시작한다.
- 긴 원문 또는 오류 메시지는 `LONG VARCHAR` 매핑을 검토한다.

## 16. 구현 기본 결정 사항

1. Goldilocks schema는 `SP`, application user는 `sp_app`으로 시작한다.
2. identity column을 기본으로 쓰고, Goldilocks 호환성 문제가 있으면 sequence로 대체한다.
3. master/ledger/risk 핵심 테이블은 FK를 즉시 적용하고, raw 대량 적재 테이블은 FK를 지연 적용한다.
4. 시각은 UTC `TIMESTAMP`와 market timezone code를 함께 저장하고 `TIMESTAMP WITH TIME ZONE`은 MVP에서 사용하지 않는다.
5. `LONG VARCHAR` JSON은 1MB 이하로 제한하고, 초과 원문은 raw storage에 둔다.
6. partitioning은 `price_bar` 5천만 row 초과 또는 월별 유지보수 필요 시 도입한다.
7. backup은 `scripts/goldilocks_backup.sh` wrapper로 native online backup을 우선 호출하고 checksum을 생성한다.
8. migration은 수동 SQL을 우선하며 Alembic autogenerate는 참고용으로만 사용한다.
9. code table은 물리 테이블로 시작한다.
10. 월간 복구 검증용 test schema 생성/삭제 절차
